约客户话术（微信）

根据是否了解客户情况选择对应场景，直接复制发送或稍作修改后使用。

场景1 不知道客户情况
XX总您好，我们最近在跟很多企业交流智能体这块时遇到一个共同问题： 智能体项目一开始都跑得挺快，但大多处于内部测试阶段，在进入生产阶段时，Agent数量一多，出现Agent环境不好维护、如遇到智能体处理任务卡慢、不知道啥原因，多个AI应用无法统一治理的问题。 我们全新发布的"AI HCI（AI超融合）"方案，主要是帮助咱们在面临Agent部署、运维管理时统一承载AI应用和Agent运行环境的。解决Agent快速部署、快速上线开启试点以及后期上线稳定运行，运维治理。 想约您找个时间交流一下，也想了解下您现在AI应用推进到哪个阶段了。
场景2 已知客户在做Agent
XX总，了解到咱们现在已经在推进Agent和大模型相关应用了。 我们了解到大家普遍会遇到一些新问题，比如： • Agent越来越多，环境越来越复杂 • 不同团队各自建设，后期很难统一管理 • AI应用上线后缺少统一运维和监控 • 模型、知识库、运行环境越来越碎片化 • GPU/模型资源不好统一调度，Token成本消耗高 我们全新发布的"AI HCI（AI超融合）"方案，主要是帮助客户统一承载AI应用和Agent运行环境，把部署、运行、运维、治理这些能力做成一体化平台。 想约您交流一下，也了解下您现在AI建设大概推进到哪个阶段了。

快速回答客户问题

客户提出常见异议时，点击展开对应应对话术，快速消除顾虑。

✅ 应对1：非线性复杂度 现在确实能跑，但随着团队扩大、Agent数量增多，管理复杂度是非线性增长的。您现在可能有2～3个Agent，但如果是20个、100个呢？每个都单独维护模型密钥、单独部署、单独处理异常？我们的平台可以对多种类型Agent进行统一运维管理（比如观测合同审核Agent审核慢原因等），为智能体运行提供可靠性底座。
✅ 应对2：平台封闭问题 每家低代码商业版平台封闭，组件工具无法复用；平台封闭导致数据孤岛，跨系统意图路由拉不通。既有开源的又有商业版低代码，技术栈不统一，管理起来也不方便。
✅ 应对3：开放不绑定 我们的平台不绑定用户使用Agent的开发平台，支持对各业务部门散落在不同低代码平台（当前支持Dify和SF-FastGPT）的Agent统一收到AI HCI上进行统一的可观测管理。
✅ 应对1：几乎不需要改造 第三方低代码平台（Dify/FastGPT）直接以导出配置文件形式纳管，版本基本无感迁移。自研高代码Agent只需要容器化（我们提供DevBox开发环境），不改业务代码，打包成OCI镜像即可接入。这是最小侵入式架构。
✅ 应对2：封闭平台迁移风险（引导客户认知） 如果使用了阿里百炼、百度千帆、字节hiagent，因涉及到各自平台知识库、数据库、记忆库技术栈封闭，无法完全迁移，各平台插件也无法导出迁移。这也是我们建议尽早统一到中立平台的原因。
✅ 应对：按架构形态适配 我们会根据ISV的架构形态做适配。比如ISV通过容器化打包部署的，都可以比较平滑地承载到AI HCI上。
💡 如果客户追问"为什么要统一到AI HCI？" 现在一、两个ISV Agent的时候问题不明显。但后面Agent越来越多，模型越来越多，数据越来越分散，运维越来越复杂。需要统一承载Agent的基础设施平台。当Agent越来越多，就一定成为运维灾难。
✅ 应对：私有化部署核心优势 理解您说的，这正是我们私有化部署的核心优势所在。AI HCI在原HCI基础上平滑演进，完全部署在您自己的机房或私有云，数据不出门，模型推理也可以选择私有化部署的模型或托管云安全算力。

覆盖产品认知、客户画像、拜访话术、问题处理与迁移场景，帮助销售与售前精准切入、高效推进。

一句话定义产品

深信服 AI 超融合（AIHCI）是面向 Agent 时代的新一代企业级 IT 基础设施，基于深信服超融合架构打造，可统一承载传统应用与 AI Agent 智能体应用，提供开箱即用、生产级可靠、开放兼容的 Agent 全栈运行环境，帮助企业快速实现 AI Agent 从试点到规模化生产落地，降低部署、运维与运营复杂度。

三大差异化核心优势

📦
Agent核心组件开箱即用

内置多种Agent运行组件（MCP/AI数据库/记忆库等）向导化部署即用

📊
高可靠 · 全链路可观测

Agent及其组件高可靠设计；Agent运行链路全局观测&LLM Token级监控，成本精确归因到部门和Agent

🔓
开放 · 不绑定

模型不绑定，兼容OpenAI、开发框架（内置高/低代码开发平台）不绑定

规模特征

2–3节点 Token 1万~10万/月

典型特征

1、IT部门本地化部署了N8N（开源产品）、dify、maxkb商业版，已经构建了10+以上智能体。

2、业务部门部署了泛微小E、金蝶AI套件如新小蝶，汉得AI中台等，私有化部署了hiagent（字节产品）。

N8N Dify商业版 MaxKB商业版 HiAgent

主要痛点

  • 资源使用情况不明，无法对部门做成本核算
  • 不同类型GPU割裂，无法统一管理
  • 10+个Agent开始出现运维管理复杂度
  • E2E优化需求涌现，响应质量难以监控
⚡ 关键购买信号 已超过10个Agent是核心购买信号。可以推AI算力网关统一纳管多品牌算力，结合AI HCI解决多Agent统一运维管理与Agent成本可视诉求。

规模特征

3节点以上 Token>10万/月 生产级Agent广泛应用

典型特征

AI部门主导整体AI业务进展，对各项产品和方案深度使用，IT部门更加关注Agent资源管理和运维类工作，有生产级agent存在，本地算力（3节点以上）或token消耗（10w/月以上）规模较大，Agent已深度定制化使用。

AI部门主导 高度定制化

主要痛点

  • 缺乏统一管理平台，Agent运营水平低
  • 预算成本制约，SLO要求越来越高
  • 开源产品功能不足，二次开发性价比低
  • 商业Agent强绑定平台，需开放兼容方案
💡 销售策略 这是AIHCI机会最大的阶段，客户需求明确。重点强调生产级可靠性、统一运营、安全合规，并对比开源方案的长期TCO劣势。

AIHCI 六讲话术框架

按顺序推进，帮助客户建立从认知到行动的完整认知链路。

01
01 · 讲认知
AI落地：从单点探索到规模化生产

Agent等场景已度过技术萌芽期，从单点探索转向规模化生产实践。不是Public AI vs Private AI的二选一，而是融合两者的"混合AI模式"。

关键话术我们不建议一上来砸钱搭大平台。去年很多公司买了算力服务器，最后都没用起来。先选一个场景跑通，见效快、风险低，跑顺了再扩。
02
02 · 讲观点
Private AI是必选项，先应用后基础设施

核心业务、客户数据、生产研发数据必须私有部署。落地路径：先跑通一个场景，AI基础设施自然跟上。

关键话术互联网公司卖云上算力，最缺的就是你行业的数据——只要用云上资源，技术上他都直接间接地拿走了你的数据。（可现场打开Coze、豆包用户协议）
03
03 · 讲挑战
企业AI落地的五大关键挑战

识别客户普遍面临的挑战，引发共鸣后再引出解决方案。

五大挑战 ① 部署复杂：硬件/AI数据库/工具链高可靠设计、部署复杂、技术门槛高
② 故障发现不及时：AI运行黑盒化，原生风险的涌现
③ 成本黑洞：算力多样，缺少有效管理手段
04
04 · 讲投入
深信服近十年"AI First"持续投入

从2016年开始，投入博士50人、研发400人、500张卡，持续投入安全GPT和AI基础设施的建设创新。

价值主张帮助用户「构建AI更简单」（SF-FastGPT），「承载AI更稳定高效」（AI超融合 + AI算力网关调度）
05
05 · 讲产品
从构建到承载，AI全栈覆盖

构建简单：AISphere统一AI门户 · SF-FastGPT低代码（0专家好效果）

承载稳定：AI超融合生产级运行环境 · AI算力调度网关治理

Agent ≠ 插件，是数字员工

Agent运行环境必须稳定可靠。各部门分散建设 = 数据孤岛 + 安全漏洞 + 运维灾难。AI基建要"全栈+解耦"，不被单一厂商绑定。

核心论点基础设施要全栈：算力、存储、平台、Agent、安全一家搞定；能力要解耦：硬件/模型/框架都不绑定

现在确实能跑，但随着团队扩大、Agent数量增多，管理复杂度是非线性增长的。您现在可能有2～3个Agent，但如果是20个、200个呢？每个都单独维护模型密钥、单独部署、单独处理异常？

我们的平台是为规模化而生的，越早引入，管理成本越低，迁移成本也越小。
几乎不需要改造。第三方低代码平台（Dify/FastGPT）直接以导出配置文件形式纳管，版本基本无感迁移。

自研高代码Agent只需容器化（我们提供DevBox开发环境），不改业务代码，打包成OCI镜像即可接入。这是最小侵入式架构，对现有业务零影响。
我们根据ISV的架构形态做适配。ISV通过容器化打包部署的，经过适配后基本可以平滑承载到AI HCI上。

当Agent越来越多、模型越来越多、数据越来越分散，就需要统一承载Agent的基础设施平台。现在一两个ISV时问题不明显，后面一定成为运维灾难。
理解您说的，这正是我们私有化部署的核心优势所在。AIHCI在原HCI基础上平滑演进，完全部署在您自己的机房或私有云，数据不出门，模型推理也可私有化部署，满足等保2.0/SOC2合规要求。

当前痛点

多实例数据混乱各部门各自部署Dify，数据隔离靠人工约定，数据管不住
API Key泄漏风险密钥直配在平台里，员工离职即高危
Token成本黑盒无法按部门/应用统计Token消耗，成本无法分摊
无操作审计出安全事件无法追溯，合规压力大

AIHCI 对应价值

工作空间三层隔离权限+数据+网关隔离，RBAC三级角色模型
密钥集中加密管理AI算力网关统一代理，AES-256加密，业务侧无需持有密钥
Token精确归因精确到工作空间/应用，成本可分摊核算
全量操作审计毫秒级UTC时间戳，满足等保2.0/SOC2
客群：有技术团队自研 LangChain / LangGraph / CrewAI / Python Agent 的企业

当前痛点

环境不一致"在我电脑能跑，到服务器各种问题"
研发持有生产密钥代码仓库和密钥混在一起，安全隐患
LLM执行黑盒出问题只能看日志，无法追溯具体哪步调用出错
工具重复开发每个Agent各造轮子，MCP工具无法复用

AIHCI 对应价值

DevBox容器化开发代码+依赖打包为不可变OCI镜像，开发即交付
密钥永不下发AI算力网关代理所有调用，研发侧用临时Token访问
四层链路追踪Session→Trace→Span→Generation，毫秒级复盘每步推理
MCP工具统一共享统一注册发现，所有Agent可复用，告别重复造轮子
客群：采购了行业软件厂商（ISV）AI应用的企业，或ISV自身希望规范交付

当前痛点

ISV掌控密钥不知道数据是否外泄，透明度低
多ISV数据孤岛各自独立，没有统一AI数据服务层
出问题等厂商上门链路不透明，买方无法自查
成本无法核实Token消耗全由ISV计费，买方无法独立核验

AIHCI 对应价值

企业掌握模型密钥ISV通过企业网关调用模型，密钥在企业手中
统一数据服务pgvector/Milvus统一管理，按需隔离或共享
ISV调用链可见走AI算力网关，Token消耗和调用链路均可见
统一认证入口网关层API Key/OAuth2/OIDC统一认证
